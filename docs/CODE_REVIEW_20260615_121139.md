## 一、安全问题

### 🔴 S1. 任何客户端都能把自己"升级"为 owner agent（垂直越权）

**位置：** `apps/wu_tanchang_api/app/endpoints/chat.py:48-57`、`async_chat.py:43-45`

`resolve_dynamic_agent` 的 owner 判定完全由请求体控制：

```python
effective_user = metadata.get("user_a_id") or metadata.get("user_id") or user_id
effective_calendar = metadata.get("user_b_id") or metadata.get("calendar_id")
is_owner = (effective_calendar is not None) and (str(effective_user) == str(effective_calendar))
```

任何攻击者只要在 `metadata` 里塞 `user_a_id=1, user_b_id=1`，就能拿到 owner agent，进而调用 `get_consultation_stats` / `list_recent_clients` / `get_client_detail`（见 S2），导出**全部**客户预聊记录与会议材料。

**修复方向与评审结论：**

> [!NOTE]
> **评审结论（2026-06-15）：暂不修改此代码。**
> **原因：** 目前 Agent API 仅在本地宿主机与 Docker 容器间私有调用，端口未对公网开放，Rails 后端作为唯一的受信调用方发起请求，安全边界受网络拓扑保护。
> **临时防护建议：** Rails 后端在向 Agent 发送请求前，必须对客户端提交的 `metadata` 进行严格清洗，剥离任何伪造的 `user_a_id`、`user_b_id`、`user_id` 或 `calendar_id` 键名，防范透传越权攻击。

**未来彻底修复方向：**
- owner 模式必须由受信通道（带签名的内部 token / 服务间 mTLS / 独立 header）显式声明，**不能**从普通客户端 metadata 推断。
- 至少在 `agent_name_override` 显式传 `"owner"` 时才允许（并配套鉴权中间件）。
- 推荐用 `Depends(require_owner_role)` 之类 FastAPI 依赖挡住 owner 工具的整个 toolset。

#### 🟡 M2. 严格 fail-safe 之后仍然可能让 `client_calendar_id = None` 的 thread 永远不被 owner 看到

**位置：** `owner_tools.py:159-165`

```python
client_calendar_id_str = str(client_calendar_id).strip() if client_calendar_id else None
# ...
if client_calendar_id_str != owner_calendar_id_str:
```

owner 端 8575c5ed 修过后 `owner_calendar_id_str` 必非空（前面有 `return []` 兜底）。**但** 客户端 thread 的 `latest_ckpt.metadata` 可能根本没有 `calendar_id`（早期数据 / 工具被绕过调用 / 元数据丢失），此时 `client_calendar_id_str` 也是 `None`：

#### 🟡 M3. `_ACTIVE_AGENTS` 没有上界 + 客户端断连可能导致悬挂

**位置：** `apps/wu_tanchang_api/agent_factory/agent.py:42-55`

`register_active_agent` 只在 `chat` 和 `chat_stream` 的正常 `finally` 里被 unregister：

- `chat.py:231/244` — `ainvoke` 包了 try/finally，正常
- `chat_stream.py:312/391-393` — `register` 在 `try` 块**之前**，`unregister` 在 `finally`。如果 `agent.astream` 抛 `asyncio.CancelledError`（客户端断 SSE），`_gen` 的 `finally` 是否运行取决于 Starlette 是否调用 `aclose()`。Starlette 的 `StreamingResponse` 在断连时通常会 `aclose` generator，所以应该会跑——但这是隐式行为。
- 真实风险点：`register_active_agent` 没人调的情况下 `_ACTIVE_AGENTS` 不增长；只要注册过、不注销，就持续占内存。当前注册方都是 try/finally，应该 OK。

**建议：**
- 给 `_ACTIVE_AGENTS` 加 LRU（如 `cachetools.LRUCache(maxsize=256)`），用 `move_to_end` 记录访问
- 在 `get_active_agent` 里加一个**过期**机制——`register` 时记时间，`get` 时检查 TTL（10 分钟），过期则返回 None

#### 🟡 M4. P5 缓存淘汰是 FIFO 不是 LRU，且会先淘汰"动态"key

**位置：** `apps/wu_tanchang_api/app/endpoints/chat.py:114-124`

```python
while len(state.agents) >= 50:
    evict_key = None
    for key in state.agents:
        if "::" in key:
            evict_key = key
            break
    if evict_key is None:
        evict_key = next(iter(state.agents))
    state.agents.pop(evict_key, None)
```

- Python 3.7+ dict 保留插入顺序，`for key in state.agents` 拿到的是**插入序**，不是 LRU
- "动态 key"（含 `::`）的优先淘汰策略意味着：所有"业务"workspace 都在含 `::` 的 key 里，而启动时塞进去的 `default` / `owner` 不含 `::` → 50 个动态 key 一来，先淘汰老的动态 agent；继续到 50 时，**开始淘汰 base key**
- 影响：基线 agent（启动预热的 `default`、`owner`）可能被运行时新编译的 owner 替换掉；下次同 base 请求触发 re-compile

**修复：**

```python
from collections import OrderedDict
# state.agents / state.agent_configs 改为 OrderedDict
# 命中即 state.agents.move_to_end(key)；淘汰时 popitem(last=False)
```

或者更简单：直接 `evict_key = next(iter(state.agents))`（FIFO，逻辑清晰），并把 cache 上限设成 `len(base_keys) + N`。

#### 🟡 M5. `compilation_lock` 懒初始化的竞态

**位置：** `apps/wu_tanchang_api/app/state.py:30-36`

```python
@property
def compilation_lock(self) -> asyncio.Lock:
    if self._compilation_lock is None:
        self._compilation_lock = asyncio.Lock()
    return self._compilation_lock
```

两个协程同时第一次访问 `state.compilation_lock` 时，可能都看到 `is None`，都新建一个 `Lock`，后写的覆盖先写的。第一个协程在 `async with` 持的是旧锁，第二个是新锁，**它们互不感知**——意味着两个协程会同时进入临界区，破坏 P4 的初衷。

实际触发概率极低（要求"两个 owner 路由请求在 1 个 await tick 内同时第一次调用"），但**修起来 0 成本**：

```python
self._compilation_lock = asyncio.Lock()  # 在 startup() 里 await 之后
```

或在第一个访问处用 `functools.cache` 风格的类级缓存。

## 二、性能问题

### 🟠 P1. `checkpoints.pkl` 会随时间无限增长 + owner 工具 O(N·M) 扫描

**位置：** `checkpointer.py:69-93`（`put` 每次写整个 dict）；`owner_tools.py:40-65`

- `DiskBackedInMemorySaver._dump_to_disk` 每次 `put`/`put_writes` 都 `pickle.dumps(整个 storage)` 写盘。`storage` 单调增长（无 TTL、无淘汰），`pickle` 时间随体量线性甚至超线性（递归 + bytes 复制），并且是**同步阻塞**全事件循环的写盘。
- `_get_client_threads` 在 owner 每次 tool 调用时遍历所有 thread + 每个 thread 调一次 `checkpointer.list(...)`，并对每个 thread 的全部 checkpoint 排序。

总成本：单次 owner 请求 ≈ O(线程数 × 线程内 checkpoint 数 × 序列化大小)。

**修复方向：**

- 给 `InMemorySaver` 加按时间或大小的 LRU 淘汰；至少 `put` 改为增量 patch + 节流（debounce）
- owner 工具把 `client_threads` 缓存到 `agent` 实例的弱引用属性，或在 `ainvoke` 期间用 `lru_cache`（key=agent_id+天数）
- 选 N 个 thread 时直接走 `checkpointer.storage` 顶层 dict 的 `started_at` 索引，不要 `list(...)` 再 sort

---

### 🟠 P2. 启动时整目录 `rmtree` + `copytree`，kb 越大越痛

**位置：** `agent_factory/utils.py:42-59`

每次冷启动把 `kb/`、`skills/`、`intake/` 和新增的多个 `workspace*/` 整个 wipe+re-copy。`kb/` 通常是几百 MB 的语料 + 索引，每次重启多花几秒到几十秒，且容易触发 `OSError: [Errno 28] No space left` 当 `runtime_dir` 配额紧张时。

**修复方向：**

- 比较 mtime/size，相同则跳过
- 或只在 `--refresh` flag 触发时全量拷，平时只做 `dirs_exist_ok` 的增量更新
- `kb/` 单独做"首次部署全量、之后增量 sync"的逻辑

---

### 🟠 P3. owner agent `max_tokens=20000` 是 default 的 25 倍

**位置：** `app/endpoints/chat.py:116`、 `config.json:8`

owner 工具的回包是结构化表格，正常 < 2k token。20k 是 10×+ 浪费，且 LLM 端"思考到 20k 才停下"会显著拉长 TTFT 和单价。

**修复方向：** owner 改成 2-4k；上限 6k 已经足够。如果担心 deep research 类长报表，让 LLM 拆轮输出。

---

### 🟠 P4. `compilation_lock` 在 async handler 中持锁做同步图编译

**位置：** `chat.py:100-132`、 `state.py:30`

`threading.Lock` + FastAPI async 协程 + 同步 `create_agent(...)`。这把锁在 LLM graph 编译期间会**阻塞事件循环**——所有其他进行中的请求都卡住。即便有 `compilation_lock`，也只是把"卡 N 个请求"变成"卡所有请求"。

**修复方向：**

- 把 `create_agent` 跑在 `await asyncio.to_thread(create_agent, ...)`
- 把锁换成 per-cache_key 的 `asyncio.Lock`，让不同 `workspace_xxx_owner` 可以并行编译
- 启动时按 `config.json` 预热所有已知 owner workspace，启动后再允许"懒加载新 workspace"——避免运行时编译

---

### 🟡 P5. 缓存 key 无界：`state.agents` 内存可被攻击者撑爆

**位置：** `chat.py:95-134`

`cache_key = f"{config_name}::{resolved_workspace}"`，`resolved_workspace` 直接由请求里的 `effective_calendar` 决定。攻击者发 10k 个不同的 `user_b_id`（在 S1 修复前甚至不需要 S1，发任意 `user_b_id` 即可），就能让进程吃下 10k 份 graph + checkpointer。每个 graph 持有模型客户端、磁盘文件路径，OOM/RSS 上升几乎确定。

**修复方向：**

- 在 S3 加白名单后这一项自动解决（白名单 key 数量有限）
- 同时给 `state.agents` 加 LRU + 容量上限（`collections.OrderedDict` + `move_to_end`）

---

### 🟡 P6. owner agent `AccountantMiddleware(max_tool_calls=20)`

**位置：** `agent.py:343`

配合 P1，每个 owner tool 调用都会做一次全 storage 扫描；20 次上限意味着单个 owner 回合可放大 20×扫描成本。

**修复方向：** 改成 4-6，并在 owner 工具里加 `if depth > 1: return "已扫描"`，避免 agent 用多次 stats 调用"探查"。

---

### 🟡 P7. owner 工具内部重复 `checkpointer.list` 排序

**位置：** `owner_tools.py:62-65, 72-73`

每个 thread 都 `list(checkpointer.list(...))` 再 `sort`——一次性 `list()` 的结果就是按时间排好的，`sort` 是冗余工作；更糟的是 `list` 内部又走 `aget_tuple` 完整反序列化。

**修复方向：** 第一次遍历时 `checkpointer.storage[tid]['']` 拿到原始 ckpt map，`max(...)` 拿最新即可，避免 `list()` 的全量反序列化。

---

### 🟢 P8. 函数内 `import` 出现在热路径

**位置：** `chat.py:43-45, 112`；`agent.py:332-336`

每次请求都重新 import。Python 会复用 `sys.modules`，所以代价不大，但延迟敏感路径里仍是微秒级浪费，且让依赖图更难看。

**修复方向：** 提到模块顶部。

## 四、顺带观察（不是这次 diff 的问题，但相关）

- `DiskBackedInMemorySaver` 用 `pickle` 持久化整个 in-memory dict：任何被 `storage` 引用的对象（包括 S5 里塞进 metadata 的 agent graph）都会被 pickle——S5 不修，等于每次 owner 请求往 checkpoint 文件写一份 graph 引用。
- owner 工具读到的 `messages` 包含 `tool_calls`，其中 `tc.get("args", {}).get("body")` 拿到的就是 `save_meeting_prep` 上传的整份材料——这是天然的"客户材料副本"留在 owner 工具上下文里，再次放大 S2 的影响面。
